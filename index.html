The Problem

Single-vuln scanners miss
the most dangerous attacks

Modern exploits don't look like a single dangerous function call. They're chains — three medium findings, each harmless alone, combined into a Critical 10.0 exploit that exfiltrates credentials before you know it happened.

🔗

Chains are invisible to traditional SAST

A single unvalidated env var scores CVSS 4.0. That same env var, three hops later writing to a config file executed next session — that's a 10.0. Traditional tools never connect those dots.

⏱

Session-crossing flows aren't tracked

Tainted data written to ~/.config/tool.json in one process and executed in the next is invisible to any scanner that stops at a single file or execution context.

🎯

Individual CVSSs tell the wrong story

Three "High" findings stay in the backlog forever. The same three findings, scored as a chain with a session-crossing persistence vector, demand immediate P0 response.

🤖

CI/CD amplifies everything

No human reviews CI output. Credential exfiltration runs silently. All environment variables — including secrets — are in scope. The same chain that scores 8.4 on a laptop scores 9.9 in CI/CD.

Coverage

All scanner types. One chain graph.

The engine merges findings across SAST, SCA, DAST, IAST, Secrets, and Container scanning into a single taint graph. Chains can cross scanner type boundaries.

SAST

Static Code Analysis

Traces tainted env vars, CLI args, and file reads through template literals and assignments to shell exec sinks. Detects cross-file and cross-session flows.

Cross-session STORED_FLOW
SCA

Software Composition Analysis

Detects vulnerable dependency versions, traces their exported methods into application code sinks. Flags malicious postinstall scripts in supply chain attacks.

Dep → App code → Exec
DAST

Dynamic / Runtime Testing

Injects payloads into HTTP parameters, monitors out-of-band callbacks (OAST). Detects SSRF to cloud metadata endpoints for credential theft chains.

HTTP → SSRF → IMDS
IAST

Interactive Instrumentation

Runtime taint propagation via OpenTelemetry hooks. Confirms SAST findings and finds chains that cross dynamic dispatch boundaries invisible to static analysis.

Runtime confirmation
Secrets

Secrets Detection

Entropy + pattern-based detection for hardcoded tokens. Traces secret values from source files and config into network exfiltration sinks. Scans git history.

Secret → Network exfil
Container

Container / IaC

Traces Docker ARG through ENV into ENTRYPOINT/CMD. Detects Helm values flowing into container command args. Flags plaintext secrets in manifests.

ARG → ENV → ENTRYPOINT
How It Works

Six phases from source code
to chain finding

The engine builds a directed taint graph across all files and sessions, then runs chain discovery to surface paths no single-sink scanner would find.

1

AST Parsing

Parse all files in the target project using tree-sitter or Babel. Extract function calls, assignments, imports, and property accesses that match source and sink patterns.

2

Source & Sink Enumeration

Catalog every untrusted input (SOURCE) and every dangerous execution point (SINK). Each scanner adapter contributes its own source/sink patterns to the unified graph.

3

Intra-file Taint Tracking

For each source, follow all assignments, template literals, and function calls within the same file. Track which variables carry tainted data toward sinks.

4

Cross-Session Flow Resolution

Detect fs.writeFile calls to sensitive config paths (STORE nodes). Match them to fs.readFile calls on the same paths (LOAD nodes). Create STORED_FLOW edges across session boundaries — this is the core capability that makes chain detection possible.

5

Chain Discovery

Run depth-first search from every SOURCE node through the taint graph. Record all paths that reach a SINK without passing through a SANITIZE node. Chains of 3+ hops with session-crossing flows are flagged as high-priority.

6

Score, Classify & Emit

Score each chain using the multiplier formula. Chains with no matching pattern in the library and a score ≥ 7.0 are flagged as zero-day candidates. Output in SARIF 2.1.0 for direct integration with GitHub Code Scanning, Snyk, and Jira.

log4j-webapp · SAST scan comparison CVE-2021-44228 · CVSS 10.0 · 3B+ devices

$ semgrep --config auto ./log4j-webapp

  Scanning 847 files...

  MEDIUM  User input flows into log statement              SearchController.java:42
          Rule: java.lang.security.audit.formatted-sql-string
          logger.info(msg) — consider sanitizing user input

  1 finding · Severity: Medium · No action required (log injection, low impact)
╚════════════════════════════════════════════════════════════════════════╝
  WHY IT MISSES: The scanner sees logger.info() as a log sink — not a code execution
  sink. It has no visibility into Log4j internals where JNDI lookup happens.
  The dangerous behaviour lives inside the library, not the application code.
  Single-hop analysis stops at the function call boundary.

codecov-bash-uploader · Secrets + SCA scan comparison Codecov Breach · Jan–Apr 2021 · 2 months undetected 
╔══ TRADITIONAL AST SCANNER (TruffleHog / GitLeaks / Bandit) ═══════════╗
$ trufflehog filesystem ./codecov.sh && bandit -r ./codecov.sh

  Scanning for hardcoded secrets...
  No secrets found.

  Scanning for security issues...
  LOW  subprocess call with shell=True                    codecov.sh:207
       Consider using subprocess module without shell=True

  0 secrets · 1 Low finding · No action required
╚════════════════════════════════════════════════════════════════════════╝
  WHY IT MISSES: Secrets scanners look for hardcoded values — API keys, tokens,
  passwords written into the file. There are none here. The script reads secrets
  from the environment at runtime, not from its own source. Static tools see
  curl at line 207 as a generic HTTP call. They cannot connect "reads all env vars
  at line 171" → "appends them at line 198" → "POSTs them at line 207" as a chain.
  Each line is innocuous alone. Together they are a complete exfiltration pipeline.

  Step 3  SINK        curl POST → [attacker-ip]/upload/v2   codecov.sh:207

This is why it ran undetected for two months across thousands of CI pipelines.
